## 一、什么是 LDAP？

LDAP，全称是 Lightweight Directory Access Protocol，轻量级目录访问协议。它的核心作用是：用统一的协议去查询、认证和管理“目录服务”中的对象。

LDAP 协议的常见端口是 TCP 389。如果使用 TLS 加密，通常叫 LDAPS，常见端口是 TCP 636。在 Active Directory 里还常见 Global Catalog，全局编录服务，端口是 3268，使用 TLS 时是 3269。

| 端口   | 协议       |
| ---- | -------- |
| 389  | LDAP     |
| 636  | LDAPS    |
| 3268 | GC LDAP  |
| 3269 | GC LDAPS |

LDAP 语境下的“目录”可以理解为一种专门存储组织结构、账号、组、计算机、权限属性等信息的**分层数据库**。

目录呈树状结构，这棵树称为 Directory Information Tree（DIT），由一个个的 Entry（条目）组成。一个 Entry 又是由一个个的 Attribute（属性）组成的。属性是 key-value 的形式，而且可以多 value，其中 key 部分被称为属性名，value 被称为属性值。

举个例子，树上的一个 Entry 中的内容可能是这样的：

```
dn: cn=Alice,ou=IT,dc=corp,dc=com
objectClass: inetOrgPerson
cn: Alice
mail: alice@corp.com
memberOf: cn=Admins,ou=Groups,dc=corp,dc=com
```

为了描述 Entry 在 DIT 上的位置，LDAP 用 Distinguished Name（DN）作为 Entry 在树上的唯一位置。

> DN 也是 Entry 属性的一部分。

DN 长这样：

```
CN=Caroline Robinson,OU=Users,DC=baby,DC=vl
```

从**右往左边**看，就是从树根一直往下走到 Entry 本身的过程。上例的含义就是，在 `baby.vl` 这个域下面的 `Users` 组织单位中，有一个 Common Name（CN）为 Caroline Robinson 的 Entry。

[[Active Directory Penetration Tester/课程内容/AD|AD]] 中的 Directory Object（目录对象）都被组织成 DIT 的形式，LDAP 就作为访问这些目录对象的协议存在。

## 二、LDAP Session

建立 LDAP Session，在 AD 场景里，其整体流程大致是：

```text
客户端
  |
  | 1. TCP 连接
  v
LDAP Server / Domain Controller
  |
  | 2. 可选：StartTLS 或直接使用 LDAPS
  |
  | 3. Bind 认证
  |
  | 4. LDAP 相关操作（查询 / 修改 / 枚举等）
  |
  | 5. Unbind / 关闭 TCP 连接
```

### 1、建立底层连接

客户端会连接 LDAP 服务器的端口。例如：

```text
LDAP   : TCP/389
LDAPS  : TCP/636
GC     : TCP/3268
GC SSL : TCP/3269
```

在 AD 中，LDAP 服务器通常就是 DC。

### 2、协商是否加密

LDAP 有两种常见加密方式：

1. 直接走 LDAPS，也就是一开始就建立 TLS 连接
2. 在 LDAP 端口先建立普通 LDAP 连接，然后发送 StartTLS 请求，把已有连接升级成 TLS

在真实 AD 环境中，是否必须加密取决于服务器配置和操作类型。

### 3、Bind

身份认证的步骤。这个后面详细讲解。

### 4、LDAP 操作

建立 LDAP Connection 之后，就可以执行 LDAP 操作。具体后面讲述。

### 5、Unbind

客户端可以发送 UnbindRequest，告诉服务器自己要结束 LDAP Session，然后关闭 TCP 连接。LDAP 的 Unbind 和普通“退出登录”不完全一样，它通常表示终止 LDAP 协议会话，不需要服务器返回响应。

### 6、抓包角度

普通 LDAP 大致会看到：

```text
TCP Three-way Handshake
LDAP BindRequest
LDAP BindResponse success
LDAP SearchRequest
LDAP SearchResultEntry
LDAP SearchResultDone
LDAP UnbindRequest
TCP Close
```

如果是 LDAPS，会先看到 TLS 握手，后面的 LDAP 内容被加密：

```text
TCP Three-way Handshake
TLS ClientHello / ServerHello / Certificate / Key Exchange
Encrypted LDAP BindRequest
Encrypted LDAP BindResponse
Encrypted LDAP SearchRequest / SearchResponse
TCP Close
```

如果是 StartTLS，则顺序是：

```text
TCP Three-way Handshake
LDAP ExtendedRequest: StartTLS
LDAP ExtendedResponse: success
TLS Handshake
Encrypted LDAP BindRequest
Encrypted LDAP SearchRequest
```

## 三、LDAP Bind

Bind 是 LDAP 协议里的**认证操作**，用来向 LDAP Server 表明“我是谁”以及“我能证明这一点”。

### 1、Bind 的三种主要类型

#### （1）Anonymous Bind

不提供任何凭证，DN 和密码都为空。服务器把你当作匿名用户，通常只能访问公开的目录信息。很多目录默认允许匿名读取部分数据。

```
DN: (空)
Password: (空)
```

#### （2）Simple Bind

最常见的方式。客户端直接发送 DN 和明文密码，服务器拿密码和条目里存储的口令做比对。

```
DN: cn=admin,dc=example,dc=com
Password: secret123
```

由于 Simple Bind 的密码是**明文传输**的，所以几乎总是应该在 TLS 加密通道上进行（即 LDAPS，636 端口，或者先用 StartTLS 把连接升级为加密）。否则密码会在网络上裸奔。

#### （3）SASL Bind

SASL（Simple Authentication and Security Layer）是一个可插拔的认证框架，支持多种机制，比如 DIGEST-MD5、**GSSAPI（详见下）**、EXTERNAL（用于客户端证书认证）等。它比 Simple Bind 更安全、更灵活，适合企业级集成，但配置也更复杂。

GSSAPI（Generic Security Services Application Programming Interface，通用安全服务应用编程接口），这是提供给 Bind 的一个抽象接口，Bind 无需关注底层真正的认证方式（Kerberos、NTLM……），只需要调用该接口就能完成认证。

那么该接口在那么多的认证方式中是如何选择的呢？

有两种模式：Negotiate（协商）和显示指定：

```
GSSAPI
├── 协商型
│   └── SPNEGO / "Negotiate"   ← 只负责选,不认证
│        └── 协商后落到 ↓
├── 真·认证机制
│   ├── Kerberos v5            ← 最常用
│   ├── NTLM (NTLMSSP)         ← 遗留,SPNEGO 的回退目标
│   ├── SPKM                   ← 公钥/证书
│   ├── LIPKEY                 ← 建在 SPKM 上
│   ├── PKU2U                  ← 微软点对点公钥
│   ├── GSS-EAP                ← 联合身份(Moonshot)
│   └── IAKERB                 ← 代理转发 Kerberos
```

显示指定很好理解，指定啥就用啥。关键是 Negotiate，在该模式下，GSSAPI 会在 Kerberos 和 NTLM 两种认证方式中选择，但是优先考虑 Kerveros，若不支持才会降级为 NTLM。

### 2、用户认证里的典型流程

在 AD 中，不像通用 LDAP 那样强制你必须先拿到完整 DN 才能 Bind。它接受几种用户记得住的标识直接拿去 Bind，然后由 LDAP Server 内部自己解析成对应账号：

- User Principal Name（UPN）：形如 `jdoe@example.com`。这是最推荐的方式。
- Down-level / NetBIOS 格式：形如 `EXAMPLE\jdoe`（域\用户名）。
- sAMAccountName：就是 `jdoe` 这个短账号名（NetBIOS 域内唯一）。

只要用户用上面这些格式之一登录，AD 就能**一次 Bind 直接验证**，完全不需要“服务账号先绑 → 搜索 → 再绑”那一整套。

那 AD 中，什么情况下还是得“两次 Bind”?

核心就一个判断，用户提供的登入标识，AD 能不能自己解析，如果不能的话，还是得退回典型的两次 Bind 场景：

1. **第一次 Bind（服务账号绑定）**：应用先用一个固定的服务账号（或管理员账号）Bind 上去，目的是获得搜索权限。
2. **搜索用户**：用用户输入的用户名（比如 `jdoe`）在目录里搜索，找到对应条目的完整 DN（比如 `uid=jdoe,ou=people,dc=example,dc=com`）。
3. **第二次 Bind（验证密码）**：用刚找到的 DN 加上用户输入的密码，再次发起 Bind。如果成功，说明密码正确，用户认证通过；如果返回错误，说明密码不对。

### 3、常见返回码

Bind 失败时，服务器会返回结果码，排查问题时很有用：

* `0` —— success，成功
* `49` —— invalidCredentials，凭证错误（密码错或 DN 不存在），这是最常见的认证失败
* `48` —— inappropriateAuthentication，认证方式不被允许（比如条目不允许这种绑定）
* `53` —— unwillingToPerform，服务器拒绝执行（常见于禁止匿名绑定或账号被禁用）

## 四、LDAP Search

LDAP 操作有很多，比如：search、compare、add……

其中，LDAP Search 是 AD 中用得最多的操作。

### 1、Search 的核心参数

一次 Search 请求，本质是告诉服务器“**从哪开始找、找多深、按什么条件筛、要返回哪些字段**”。

具体由这几个参数决定：

- Base DN：从目录树的哪个节点开始往下找。比如 `ou=people,dc=example,dc=com` 表示只在 people 这个分支里找。Base 选得越精确，搜索越快、范围越小。
- Scope：决定从 Base 往下找多深，三个取值：
	- `baseObject(base)`：只看 Base 这一个 Entry 本身，不往下
	- `singleLevel(one)`：只看 Base 的直接子节点，不含孙子
	- `wholeSubtree(sub)`：Base 及其下所有层级，全找（最常用）
- Filter：决定“什么样的条目算匹配”，很重要，下面整节专门讲
- Attributes：指定每个匹配条目要带回哪些字段。不写默认返回所有用户属性；只想要特定字段就列出来，比如只要 `cn mail`。有两个特殊值得知道：`*` 表示所有普通属性，`+` 表示所有操作属性（operational attributes，如 `memberOf`、`whenCreated` 这类系统维护的字段，很多时候不显式要就不返回）
- 其他控制项：比如 `sizeLimit`（最多返回几条）、`timeLimit`（最长搜多久）、以及分页控制（下面会提）。

一条典型的 `ldapsearch` 命令把这些拼在一起长这样：

```bash
ldapsearch -H ldaps://ad.example.com \
  -D "svc-account@example.com" -W \         # 用谁的身份绑定
  -b "ou=people,dc=example,dc=com" \        # Base DN
  -s sub \                                  # Scope
  "(&(objectClass=user)(mail=jdoe@example.com))" \  # Filter
  cn mail memberOf                          # 要返回的属性
```

### 2、Filter 语法

Filter 是一套有特定规则的表达式语法，定义在 RFC 4515。

#### （1）基本结构

```
(属性 操作符 值)
```

比如：

```
(cn=John Doe)         名字正好是 John Doe
(uid=jdoe)             uid 是 jdoe
(objectClass=user)     是 user 类的条目
```

#### （2）操作符

![[file-20260611095932802.png]]

>[!attention]
> LDAP 没有单独的 `>` 和 `<`，只有 `>=` 和 `<=`。

通配符 `*` 在值里表示“任意字符序列”，可以放在不同位置：

```
(cn=John*)        以 John 开头
(cn=*Doe)         以 Doe 结尾
(cn=*oh*)         中间含 oh
(mail=*@example.com)   邮箱属于 example.com 域
(mail=*)          只要有 mail 属性(存在性判断)
```

#### （3）逻辑运算符

```
&    AND(与)
|    OR(或)
!    NOT(非)
```

逻辑运算符采用的是前缀（波兰式）记法，结构是：

```
(逻辑运算符(条件一)(条件二))
```

例如：

```
(&(objectClass=user)(mail=jdoe@example.com))
表示：既是 user,邮箱又是这个
(|(department=Sales)(department=Marketing))
表示：部门是 Sales 或 Marketing
(!(objectClass=computer))
表示：不是 computer 的条目
```

**`&` 和 `|` 可以跟两个以上条件**，挨个排开就行：

```
(&(objectClass=user)(department=Sales)(mail=*))
表示：是 user,且属于 Sales,且有邮箱(三个条件全满足)
```

逻辑运算符可以任意嵌套，每个 `(...)` 都是一个完整的“条件块”，可以整块塞进另一个运算符里：

```
(&(objectClass=user)(|(department=Sales)(department=Marketing)))
```

读法：是 user，并且（部门是 Sales 或 Marketing）。

按照下述方式拆开看结构会更清楚：

```
&                                   
   (objectClass=user)              
   (|                              
      (department=Sales)
      (department=Marketing)
   )
)

```

#### （4）AD 特有：UAC

有个 AD 专属的高级用法：`userAccountControl`（UAC） 

在 AD 中，一个账号对象有很多的不同的配置、类型和状态，这些信息如果再做成其对应 Entry 的一个个的属性，就会出现使得 Entry 变得臃肿，并且也不方便存储。因此，采用 `userAccountControl` 这个特殊的属性来表示。

关于账号的配置、类型和状态都会被一个位字段表示，常见的：

| 十进制        | 十六进制        | 标志名                              | 含义                                      |
| ---------- | ----------- | -------------------------------- | --------------------------------------- |
| `2`        | `0x0002`    | `ACCOUNTDISABLE`                 | 账号被禁用                                   |
| `8`        | `0x0008`    | `HOMEDIR_REQUIRED`               | 需要主目录                                   |
| `16`       | `0x0010`    | `LOCKOUT`                        | 账号被锁定                                   |
| `32`       | `0x0020`    | `PASSWD_NOTREQD`                 | 不要求密码                                   |
| `64`       | `0x0040`    | `PASSWD_CANT_CHANGE`             | 用户不能改密码                                 |
| `128`      | `0x0080`    | `ENCRYPTED_TEXT_PWD_ALLOWED`     | 允许可逆加密存储密码                              |
| `512`      | `0x0200`    | `NORMAL_ACCOUNT`                 | 普通用户账号(最常见)                             |
| `2048`     | `0x0800`    | `INTERDOMAIN_TRUST_ACCOUNT`      | 域间信任账号                                  |
| `4096`     | `0x1000`    | `WORKSTATION_TRUST_ACCOUNT`      | 工作站/成员服务器计算机账号                          |
| `8192`     | `0x2000`    | `SERVER_TRUST_ACCOUNT`           | **域控制器账号**                              |
| `65536`    | `0x10000`   | `DONT_EXPIRE_PASSWORD`           | 密码永不过期                                  |
| `262144`   | `0x40000`   | `SMARTCARD_REQUIRED`             | 要求智能卡登录                                 |
| `524288`   | `0x80000`   | `TRUSTED_FOR_DELEGATION`         | 信任此账号用于委派(无约束委派,安全敏感)                   |
| `1048576`  | `0x100000`  | `NOT_DELEGATED`                  | 该账号敏感,不可被委派                             |
| `2097152`  | `0x200000`  | `USE_DES_KEY_ONLY`               | 仅用 DES 加密类型                             |
| `4194304`  | `0x400000`  | `DONT_REQ_PREAUTH`               | 不要求 Kerberos 预认证(易受 AS-REP Roasting 攻击) |
| `8388608`  | `0x800000`  | `PASSWORD_EXPIRED`               | 密码已过期                                   |
| `16777216` | `0x1000000` | `TRUSTED_TO_AUTH_FOR_DELEGATION` | 约束委派(协议转换)                              |

`userAccountControl` 做的就是把该账号拥有的位字段进行累加，将最终的数字作为自己的 Value 存在。

常见的查询方法：

```
# 所有被禁用的账号
(userAccountControl:1.2.840.113556.1.4.803:=2)

# 所有「启用」的普通用户(普通账号 且 未禁用)
(&(objectClass=user)(objectCategory=person)(!(userAccountControl:1.2.840.113556.1.4.803:=2)))

# 密码永不过期的账号(安全审计常查)
(userAccountControl:1.2.840.113556.1.4.803:=65536)

# 所有域控
(userAccountControl:1.2.840.113556.1.4.803:=8192)

# 不要求 Kerberos 预认证的账号(AS-REP Roasting 风险点)
(userAccountControl:1.2.840.113556.1.4.803:=4194304)

# 配置了无约束委派的账号(高危,横向移动常用入口)
(userAccountControl:1.2.840.113556.1.4.803:=524288)
```

那串 OID `1.2.840.113556.1.4.803` 是 AD 的 **bitwise AND** 匹配规则（还有 `...804` 是 bitwise OR）：

```
1                    → iso(ISO 根)
1.2                  → iso member-body(各国标准机构)
1.2.840              → 美国(ANSI 分配给 US 的号)
1.2.840.113556       → Microsoft(微软自己的企业号,这一段就代表"微软")
1.2.840.113556.1     → 微软下的 AD
1.2.840.113556.1.4   → AD 的「控制 / 匹配规则等」分支
1.2.840.113556.1.4.803 → 具体到「bitwise AND 匹配规则」这一个东西
```

常见的 OID：

| OID | 名称 | 用途 |
|-----|------|------|
| `1.2.840.113556.1.4.803` | `LDAP_MATCHING_RULE_BIT_AND` | 按位 AND — 属性中**包含**指定 bit 时才匹配 |
| `1.2.840.113556.1.4.804` | `LDAP_MATCHING_RULE_BIT_OR` | 按位 OR — 任意 bit 匹配即可 |
| `1.2.840.113556.1.4.1941` | `LDAP_MATCHING_RULE_IN_CHAIN` | 递归链匹配 — 常用于嵌套组 |

为什么不直接写成：

```
(userAccountControl=512)
```

这样的等值匹配呢？

前面提到，属性 `userAccountControl` 的值是多个位字段累加后的结果，因此用等值匹配会漏掉很多的信息。

而：

```
(属性名:OID:=值)
```

这是一种扩展匹配语法，匹配的规则就是由指定的 OID 决定的。

`1.2.840.113556.1.4.803` 这个 OID 绑定了一段具体的比较逻辑：把属性的整数值，和你给的整数值，做“按位与”。如果结果等于你给的那个值，就算匹配。用 `8192` 举例，服务器内部做的是:

```
给定值 = 8192

8192 AND 8192 = 8192 ,等于给定值 → 匹配 ✓

8192+512 = 8704
8704 AND 8192 = 8192 ,等于给定值 → 匹配 ✓

512 AND 8192 = 0,不等于 8192 → 不匹配 ✗
```

### （5）特殊字符转义

如果属性值本身含有 `*` `(` `)` `\` 或 NUL 这些字符，必须转义成 `\` + 两位十六进制，否则会被当成语法符号：

```
*  →  \2a
(  →  \28
)  →  \29
\  →  \5c
NUL → \00
```

例：要精确匹配名字 `John*Doe`（名字里真有个星号）：

```
(cn=John\2aDoe)
```

#### （6）常见 Filter 速查

| 目标 | LDAP Filter |
|------|-------------|
| 所有用户 | `(&(objectCategory=person)(objectClass=user))` |
| 所有组 | `(objectClass=group)` |
| 所有计算机 | `(objectCategory=computer)` |
| 域控 | `(&(objectCategory=computer)(userAccountControl:1.2.840.113556.1.4.803:=8192))` |
| 禁用账户 | `(userAccountControl:1.2.840.113556.1.4.803:=2)` |
| 有 SPN 的账户 | `(servicePrincipalName=*)` |
| 描述不为空 | `(description=*)` |

> `objectCategory` 通常是对象主要类别（搜索效率较好），`objectClass` 表示对象类继承链。实践中常组合使用 `(&(objectCategory=person)(objectClass=user))` 来精确匹配用户对象。

### 3、分页

AD 默认对单次搜索返回的条目数有上限（经典是 **1000 条**，`MaxPageSize` 策略控制）。如果你搜出来的结果可能超过这个数，**不分页就会被截断**。

>[!tip]
> 只拿到前 1000 条是正常现象，并不报错。若不知道有这个最大限制存在，就容易觉得搜索结果就 1000 条，容易踩坑。

解决办法是用 **paged results control**（分页控制，OID `1.2.840.113556.1.4.319`），一页页拉：

```
ldapsearch ... -E pr=1000/noprompt "(filter)"
```

写代码时（`ldap3`、JNDI 等）对应的是“simple paged results”控件，循环带 cookie 翻页直到取完。处理 AD 大规模查询时这几乎是必备的。

### 4、PowerShell -Filter 与 -LDAPFilter

#### （1）介绍

`-Filter` 和 `-LDAPFilter` 是 PowerShell 的 **ActiveDirectory 模块**（`Get-ADUser`、`Get-ADComputer`、`Get-ADObject` 等命令）里两个用来筛选的参数，它们最终都通向 LDAP 搜索，但**写法和心智模型完全不同**：

* `-LDAPFilter`：直接写原生 LDAP filter 字符串，就是前面一直在讲的那套 `(&(...)(...))` 前缀记法语法，原封不动塞进去。
* `-Filter`：写 PowerShell 风格的表达式（类似 PowerShell 的比较语法，`-eq`、`-and`、`-like` 这些），由模块在背后翻译成 LDAP filter 再发给域控。

同一个查询（筛“启用状态、且部门是 Sales 的用户”），两种写法对照：

`-LDAPFilter`：

```powershell
Get-ADUser -LDAPFilter "(&(objectClass=user)(department=Sales)(!(userAccountControl:1.2.840.113556.1.4.803:=2)))"
```

`-Filter`：

```powershell
Get-ADUser -Filter "Enabled -eq 'True' -and Department -eq 'Sales'"
```

注意几个差异点：

* `-Filter` 里用的是属性的“PowerShell 友好名”（如 `Enabled`），而不是原始 LDAP 属性名。`Enabled` 其实是模块帮你封装的，它背后对应的正是 `userAccountControl` 的禁用位。你不用自己写那串 `:1.2.840.113556.1.4.803:=2`，模块替你翻译了。
* `-LDAPFilter` 里你必须用真实 LDAP 属性名（`department`、`userAccountControl`），而且得自己处理位匹配那套。

#### （2）各自的优劣

`-Filter` 的好处：可读、符合 PowerShell 习惯、复杂查询容易写、错误信息更容易看懂等

```powershell
Get-ADUser -Filter "Name -like 'John*'"
Get-ADUser -Filter "mail -eq 'jdoe@example.com'"
Get-ADUser -Filter {Department -eq "Sales"}   # 也可用脚本块写法
```

| Filter          | Meaning                  |
| --------------- | ------------------------ |
| -eq             | Equal to                 |
| -le             | Less than or equal to    |
| -ge             | Greater than or equal to |
| -ne             | Not equal to             |
| -lt             | Less than                |
| -gt             | Greater than             |
| -approx         | Approximately equal to   |
| -bor            | Bitwise OR               |
| -band           | Bitwise AND              |
| -recursivematch | Recursive match          |
| -like           | Like                     |
| -notlike        | Not like                 |
| -and            | Boolean AND              |
| -or             | Boolean OR               |
| -not            | Boolean NOT              |

对写惯 PowerShell 的人很顺手，日常简单查询基本都用它。

`-Filter` 的坑：

* 它有自己的**翻译层**，某些复杂条件翻译不了或翻译得别扭，尤其是**位运算**。
* 引号、变量插值容易出问题。比如用变量时：
```powershell
$dept = "Sales"
Get-ADUser -Filter "Department -eq '$dept'"   # 变量要插进字符串，注意引号嵌套
```

`-LDAPFilter` 的好处：精确、强大、可移植

* 能写任何合法的 LDAP filter，包括 `-Filter` 搞不定的位匹配、扩展匹配规则、复杂嵌套。
* 它就是标准 LDAP 语法，所以你在 `ldapsearch`、Python `ldap3`、其他工具里调好的 filter，可以原样复制到 `-LDAPFilter`，跨工具通用。
* 想精确控制发给域控的到底是什么查询时，用它最可靠（没有翻译层的不确定性）。

`-LDAPFilter` 的代价：门槛稍高，需要先了解 LDAP 语法，复杂的查询语句需要写很长。

## 五、AD 中的组嵌套关系

### 1、什么是组嵌套？

在 Active Directory 中，组嵌套指的是一个组成为另一个组的成员。例如：

```
Domain Admins
  └── IT-Admins (组)
        └── Helpdesk-Tier2 (组)
              └── 张三 (用户)
```

这里张三并不是 `Domain Admins` 的直接成员，但通过 `IT-Admins → Helpdesk-Tier2` 的嵌套链，他**间接**获得了 `Domain Admins` 的权限。这种间接的成员关系是组嵌套的核心。

AD 支持的组类型（安全组）按作用域分为：

- **Universal**（通用组）
- **Global**（全局组）
- **Domain Local**（域本地组）

它们之间存在嵌套规则（如 AGDLP 模型：Account → Global → Domain Local → Permission），实际环境中嵌套层级往往很深。

### 2、为什么要关注组嵌套现象？

#### （1）隐藏的权限路径

直接查询 `Domain Admins` 的成员可能只看到几个账户，但深层嵌套会隐藏大量"间接管理员"。一个看似普通的低权限组，可能因为多层嵌套最终拥有高权限。攻击者寻找的正是这些**非显而易见的提权路径**。

#### （2）攻击面分析

如果你拿下一个普通用户，递归分析它所属的所有组（包括嵌套），能发现它实际能访问哪些资源、是否间接具备特权。

>[!tip] BloodHound 的核心逻辑
> BloodHound 之所以强大，本质就是把组嵌套、ACL、会话等关系建模成图，自动找出"从当前身份到 Domain Admin 的最短路径"。`MemberOf` 边正是组嵌套关系的体现。

### 3、如何递归查询组嵌套

#### （1）LDAP_MATCHING_RULE_IN_CHAIN（最高效）

AD 提供了一个特殊的匹配规则 OID `1.2.840.113556.1.4.1941`，它会让 LDAP 服务器**在服务端自动递归**遍历嵌套链。

查询某用户所属的所有组（含嵌套）：

```
(member:1.2.840.113556.1.4.1941:=CN=张三,OU=Users,DC=corp,DC=com)
```

查询某组的所有成员（含嵌套展开）：

```
(memberOf:1.2.840.113556.1.4.1941:=CN=Domain Admins,CN=Users,DC=corp,DC=com)
```

这种方式的优点是一次查询解决，缺点是开销大、深层嵌套可能较慢。

#### （2）PowerShell（RSAT / AD 模块）

```powershell
# 递归列出某组的所有成员（含嵌套）
Get-ADGroupMember -Identity "Domain Admins" -Recursive

# 递归列出某用户所属的所有组
Get-ADPrincipalGroupMembership "张三" | Select Name
```

`-Recursive` 参数会自动展开嵌套。

#### （3）ldapsearch（Linux）

```bash
ldapsearch -x -H ldap://dc.corp.com -D "user@corp.com" -w 'password' \
  -b "DC=corp,DC=com" \
  "(memberOf:1.2.840.113556.1.4.1941:=CN=Domain Admins,CN=Users,DC=corp,DC=com)" \
  sAMAccountName
```

#### （4）BloodHound（图分析）

实战中最常用。SharpHound 收集数据后导入 Neo4j，用 Cypher 查询或内置路径分析直接可视化嵌套关系：

```cypher
MATCH p=(u:User)-[:MemberOf*1..]->(g:Group {name:"DOMAIN ADMINS@CORP.COM"})
RETURN p
```

`*1..` 表示任意深度的嵌套。

### 4、实践注意点

- **避免无限循环**：AD 理论上不允许循环嵌套，但脚本自己递归时仍要做去重/访问记录。
- **`1941` 规则的代价**：在大型域里对深层嵌套用 `1941` 可能很慢，且容易被监控系统注意到。
- **primaryGroupID**：用户的主组（通常 Domain Users）不通过 `memberOf` 体现，递归时容易遗漏，需单独处理。
- **跨域/跨林**：Universal 组和信任关系会让嵌套跨越域边界，分析时要考虑全局编录（Global Catalog）。
