# Claude 提示词设计

>[!tip]
>Gemini 的提示词设计技巧也是如此。唯独 GPT 有所不同，可见文章 [[GPT 提示词设计]]。

## 模板

```xml
<documents>
  <document index="1">
    <source>来源名称</source>
    <document_content>正文……</document_content>
  </document>
</documents>

<role> 角色设定 </role>

<context> 背景与动机 </context>

<instructions> 分步骤要求 </instructions>

<examples>
  <example>...</example>
</examples>

<input> 实际要处理的数据 </input>

<output_format> 输出结构要求 </output_format>
```

其中，`<input>` 标签可以根据不同的任务切换成更贴合任务的标签。比如在代码审计场景，可以改成 `<code>`；讲解知识点的时候，可以替换成 `<topic>`。而且，可以不局限于“用一个标签表达”，可以多标签展示任务。

## 总体原则

- **提示词要“清晰明确”**：想要"超出预期"的表现，就直接说出来，不要指望模型从模糊指令里自己揣摩。文档举例：与其说"做一个分析仪表盘"，不如说"做一个分析仪表盘，尽可能包含相关的功能与交互，不要停留在基础版本"。
- **正向指令优于负向指令**：与其说"不要用 markdown"，不如说"用连贯流畅的段落来写"。告诉模型"做什么"比"别做什么"更可控。
- **关于 XML 标签**：命名要保持一致、有描述性；有层级关系时就嵌套（比如多份文档用 `<documents>` 包住，每份是 `<document index="1">`）；可以直接在内容中指代 XML 标签，例如：“请总结 `<source_text>` 标签中的内容，并排除 `<blacklist>` 中的词汇。”（注意，指代的时候用反引号包裹标签，以免发生标签混乱）。
- **“列示例”是最可靠的引导手段之一**：建议放 3–5 个示例，并且要满足三个标准：贴近真实用例、覆盖边界情况、用 `<example>` 标签包裹。示例要"多样"很重要，否则模型会学到你没想让它学的潜在模式。
- 文档/资料放最前面：官方文档中说这样在复杂多文档场景下回答质量最多能提升约 30%。同时可以让模型"先摘录文档中相关的原文片段，再开始作答"，帮它从噪声里聚焦。

## 高级技巧

官方给的防幻觉：

```xml
<investigate_before_answering>
在对任何代码下安全结论前，必须先阅读相关文件。
绝不臆测未打开过的代码，也绝不为了让示例"看起来有漏洞"而编造行号或函数名。
若证据不足，明确说明"需要进一步查看 X 文件"，而不是猜测。
</investigate_before_answering>
```

## 示例

### 示例一

让 Claude 担任代码安全审计员（带防幻觉护栏）

```
<role>你是一名应用安全（AppSec）工程师，专做安全代码审计。</role>

<investigate_before_answering>
在对任何代码下安全结论前，必须先阅读相关文件。
绝不臆测未打开过的代码，也绝不为了让示例"看起来有漏洞"而编造行号或函数名。
若证据不足，明确说明"需要进一步查看 X 文件"，而不是猜测。
</investigate_before_answering>

<instructions>
审计 <code> 时：
1. 按 OWASP Top 10 维度逐项核查（注入、认证、敏感数据暴露等）。
2. 每发现一处问题，用如下结构报告：
   - 位置（文件:行号）
   - 漏洞类型与 CWE 编号
   - 可被利用的具体路径（数据从哪进、到哪触发）
   - 修复建议（给出安全的写法，而非笼统的"做好校验"）
3. 用连贯的段落撰写总体风险评估，不要堆砌过短的要点。
4. 只报告你有实际代码证据支撑的问题；不确定的列入"需进一步确认"。
</instructions>

<code>
{在此粘贴待审计代码}
</code>
```

### 示例二

```xml
<role>
你是一位网络安全课程的资深讲师，擅长把底层原理讲透、把抽象机制讲具体。
你的目标是让学习者"理解为什么"，而不只是"记住是什么"。
</role>

<context>
本次讲解面向 {目标受众，如：信息安全专业大二学生 / 转行的开发工程师}，
他们已具备 {已有基础，如：基本网络协议与编程概念}，
但对 {本知识点} 还不熟悉。
讲清楚原理本身比罗列防御清单更重要——理解了机制，防御措施自然能推导出来。
</context>

<topic>
本次教学知识点：{在此填入，如：TCP SYN Flood 的攻击原理}
</topic>

<instructions>
请按以下顺序组织讲解：
1. 一句话定位：先用一句通俗的话说清楚这个知识点"本质上在解决/利用什么"。
2. 前置原理：讲清楚理解它所必需的底层机制（比如协议握手、内存布局、权限模型）。
   如果某个前置概念学生可能不熟，先用一个生活类比帮他建立直觉，再回到技术细节。
3. 原理拆解：分步骤说明它是如何运作的，让因果链清晰可见——
   "因为 A，所以攻击者能 B，进而导致 C"。
4. 最小演示：给出一个尽量简化的示例（伪代码、协议交互图或带注释的片段），
   只保留体现原理的核心部分，去掉无关细节。
5. 防御与边界：解释为什么常见的防御手段能奏效（对应到前面的原理），
   以及它们的局限在哪里。
6. 检验理解：最后提出 2–3 个由浅入深的思考题，帮助学习者自检。

撰写要求：
- 用连贯流畅的段落讲解，必要时用编号步骤体现顺序，但不要把内容碎成一堆短要点。
- 出现专业术语时，第一次出现要给出简短解释。
- 讲到关键因果处可以追问"为什么"，引导学生主动推理，而不是直接喂结论。
</instructions>

<constraints>
- 演示代码/命令仅用于阐明原理，保持最小化，不提供可直接用于实际攻击的完整武器化工具或目标参数。
- 涉及攻击技术时，始终从"理解机制以便防御"的教学视角展开。
- 如果学习者基础不足以理解某一步，主动指出需要先补哪个前置知识。
</constraints>
```

## 实话实说

```xml
<instructions> 分步骤要求 </instructions>

<examples>
  <example>...</example>
</examples>

<output_format> 输出结构要求 </output_format>
```

上述三个标签，对人来说特别不友好。如果设计得不好，反而会拉低模型的能力。

因此，这一部分推荐让 AI 自己设计。

还有个更便捷的方案，直接让 Claude 根据官方文档以及你要完成的任务，自己设计出一份优质提示词，大致模板：

```xml
<document>
https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices
https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/prompting-claude-opus-4-8
</document>
<role>
你是一位专业的 AI 提示词设计专家，你设计的提示词能最大化地展现 AI 的能力
</role>
<task>
描述你要这份提示词去完成的任务
</task>
```
