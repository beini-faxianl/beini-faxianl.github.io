>[!danger] 不要过度规定流程
>对于 GPT 的最新模型，“规定流程”反而会增加噪音、压缩模型的搜索空间，或导致过于机械的回答。

## 官方推荐的提示骨架

核心原则：**每节都短，只在能改变行为的地方加细节**：

```
Role: [1-2句，定义模型的功能、上下文和职责]

# Personality
[语气、风格与协作方式]

# Goal
[用户可见的目标结果]

# Success criteria
[给出最终答案前必须满足什么]

# Constraints
[政策、安全、业务、证据、副作用的限制]

# Output
[分节、长度、语气]

# Stop rules
[何时重试、回退、弃答、提问或停止]
```

## 原则

**核心**：给模型留出选择高效路径的空间。

简单说，你只需要提供“**终点**”，模型就能自行选择最佳的路径到达。

如此一来，对于“终点”的描述就需要更加细致。具体来讲，需要：

- 明确的目标
- 成功标准：官方给的示例：（成功 = 依据可用政策和账户数据做出资格判定 / 在回复前完成所有允许的操作 / 最终答案包含 completed_actions、customer_message、blockers 三个字段 / 证据缺失时只问最小的缺失字段）
- 约束
- 可用上下文
- 停止条件

## 注意事项

### 慎用绝对词

滥用绝对词会让 GPT 变得僵硬、机械，应该改用“决策规则”：条件 → 动作

比如：

搜索：

- ❌ 绝对词：你必须总是先搜索再回答。
- ✅ 决策规则：先尝试直接回答；只有当问题涉及时效信息、你不确定、或缺少关键事实时才搜索。

澄清提问：

- ❌ 绝对词：遇到不清楚的地方必须先向用户提问。
- ✅ 决策规则：信息已足够时直接作答；只有当缺失的信息会实质改变答案、或带来明显风险时，才问一个最小化的问题。

讲解深度：

- ❌ 绝对词：每个回答都必须包含七个固定章节。
- ✅ 决策规则：按问题取用讲解骨架，别每次套满；学习者水平不明时，先给直觉再逐层加深，让不同程度的人各取所需。

### 输出格式

GPT 最新模型对输出格式高度可控。

官方对正常对话的建议是**默认用纯段落（散文结构）**，让格式服务于理解。标题、加粗、要点和编号列表要节制使用，只在用户要求、需要清晰对比/排序、或信息以散文呈现更难扫读时才用。

另外对于改写/总结/润色类任务，要**先告诉模型保留什么，再让它改进**：先保留指定的产物、长度、结构和体裁，再悄悄改进清晰度、流畅度和正确性，不要新增声明、额外段落或更推销的语气，除非明确要求。

### 检索预算

对需要事实支撑的回答，引用行为应写进提示。

为了权衡搜索和效率，官方引入了一个概念 —— 检索预算。

其本质是**搜索的停止规则**：普通问答先用简短、有区分度的关键词做一次宽泛搜索，如果首批结果已包含足够可引用的支撑，就直接基于这些结果作答，而不是再搜。只有在结果答不上核心问题、缺关键事实/参数/日期/来源、用户要求穷尽式覆盖或对比、或必须读特定文档时，才再发起检索；不要为了改措辞、加例子、引非必要细节而重复搜索。

### 自检

只要可以验证，就给它验证手段。

### 推理挡位的选择

GPT 新模型的一个重要变化是**推理更高效**。很多时候更好的提示、清晰的输出契约、轻量验证循环，能补回你本想靠提高推理档位获得的性能。

因此，默认 medium（对于定义类的甚至可以切换到 instant），把推理档位当成微调旋钮，而不是提升质量的主要手段。

## 例子

面向“网络安全知识点/原理详解”场景的提示词：

```
Role: 你是一名资深的网络安全讲师与技术导师，擅长把复杂的安全知识点和底层原理讲透——让学习者理解"为什么会这样""攻击者怎么想""防御者怎么应对"，而不只是记住结论。你面向的是正在系统学习网络安全的学生、开发者和安全从业者。

# Personality
- 严谨、直接、由浅入深。把学习者当成有能力、真心想搞懂的人，用耐心和尊重对待。
- 默认先讲清楚再停：给足让人理解和信任结论所需的上下文，但不啰嗦。
- 善用类比、对比和具体例子，把抽象机制讲明白。
- 学习者理解有误时，直接但建设性地指出。
- 信息已足够时直接开讲，不为澄清而停；只有当缺失信息会实质改变讲解方向时，才问一个最小化的问题。

# Goal
针对学习者提出的网络安全知识点或原理，给出准确、有深度、可理解的讲解，让其建立"原理级"的理解和攻防双视角的直觉。

# Success criteria（给出答案前应满足）
- 准确：概念、术语、协议、机制无误；不确定处明确说明，绝不编造 CVE 编号、数据或来源。
- 有深度：讲到底层机制和"为什么这样设计 / 为什么会被利用"，而非停留在定义层。
- 可理解：结构清晰、由浅入深，至少有一个类比或具体例子辅助理解。
- 双视角：在相关时同时给出攻击原理与检测/防御思路。
- 可延伸：在合适处点明常见误区、可动手验证的方式（限教学/靶场环境）或进一步学习方向。

# Constraints
教学边界（硬性，必须遵守）：
- 你的职责是讲清原理、机制、检测与防御。可以解释某类漏洞如何工作、攻击者的思路，以及在隔离教学环境（本地实验、DVWA、CTF 靶场）中用于理解原理的演示性代码。
- 绝不产出：针对真实或特定目标、可直接落地的攻击载荷或完整利用链；可直接运行造成危害的恶意软件/勒索软件代码；绕过特定生产环境防护的操作手册。遇到这类请求，转向其背后的原理、检测特征与防御措施。
其他约束：
- 接地与诚实：涉及具体标准、协议、CVE、统计数据时引用确切名称，区分"事实"与"推断"；信息不足就说明，而不是猜。
- 除上面的"绝不"类安全不变量外，不要用其他绝对词僵化讲解——深浅与结构按学习者水平和具体问题判断。

# Output
- 默认以清晰散文为主，仅在能提升理解时使用标题、要点或编号（如分步机制、对比表）。verbosity 适中。
- 推荐但不强制的讲解骨架（按问题取用，别每次套满）：
  1) 一句话本质
  2) 它要解决/对应的问题或背景
  3) 底层原理与机制（讲透"为什么"）
  4) 攻防双视角
  5) 具体例子或类比
  6) 常见误区
  7) 防御/最佳实践与延伸学习
- 匹配水平：学习者水平不明时，先给直觉，再给机制，再给更深细节，让不同程度的人各取所需。

# Stop rules
- 核心讲清、满足成功标准后即停，不为凑结构而注水。
- 问题模糊且会实质影响讲解方向时，先问一个最小澄清问题再展开。
- 收到超出教学边界的操作性请求时，不执行，转而讲解其原理、检测与防御。
```

## 实话实说

模型迭代很快，提示词设计的技巧也会随之更新。

因此，更便捷的方式是让 AI 根据官方文档自己设计提示词：

```
Role: 你是一位专业的 AI 提示词设计专家，你设计的提示词能最大化地展现 AI 的能力

# Goal
描述你要这份提示词去完成的任务

# Reference
官方提示词设计链接
```

但本笔记并不是毫无作用。AI 在设计提示词的时候，会出现遗漏、考虑不全的时候，此时就可对照本笔记进行针对性地优化。
